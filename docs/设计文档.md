## 1. 本方案解决什么问题

当前大语言模型在实用化部署中普遍面临以下**七个深层问题**，传统方法往往只针对其中一两点“打补丁”，难以系统性根治。本方案提供一套统一架构，同时解决这些问题。

| 问题 | 具体表现 | 本方案如何解决 |
|------|----------|----------------|
| **① 无限自指与递归瘫痪** | 要求模型“解释自己的思考过程”时，可能陷入元-元-元描述的死循环，耗尽上下文窗口 | 时间解耦 + 截断控制器强制深度限制 |
| **② 过程黑盒与元认知缺失** | 模型给出答案却无法说明“依据什么得出”，高风险场景无法建立信任 | 第二层自描述小模型对冻结快照进行事后分析，输出自然语言推理路径 |
| **③ 价值观可篡改与越狱** | 精心构造的提示词可绕过RLHF安全护栏；模型权重若被窃取可随意微调去安全限制 | 硬编码注意力偏置 $B_{\text{core}}$，不依赖模型权重，物理上无法绕过 |
| **④ 注意力不可控与有害生成** | 模型可能过度关注偏见或虚假信息，缺乏外部“方向盘”实时纠偏 | 第三层元注意力控制器提供软约束引导，在推理前注入偏置 |
| **⑤ 用户定制与安全边界矛盾** | 允许用户调节风格参数会带来滥用风险，完全禁止又牺牲体验 | 有限外部接口 + 安全投影，确保用户操作在核心价值观划定的安全子空间内 |
| **⑥ 审计不可行与证据链缺失** | 输出争议时无法回溯“模型当时依据了什么”，日志可被单方篡改 | 隔离审计日志 + 哈希链 + Merkle树，支持第三方独立验证 |
| **⑦ 审计日志数据爆炸** | 详细日志在百万级并发下产生TB级日增量，存储成本无法承受 | 语义压缩 + 列式存储 + 块级聚合，压缩比超30倍 |

## 2. 设计哲学：觉悟者认知调控的工程映射

本架构的层次设计借鉴了佛教修行中“止观双运”的认知调控原理，并将其转化为可工程化的计算组件：

| 觉悟者概念 | 通俗解释 | 架构对应 | 实现形式 |
|------------|----------|----------|----------|
| **根本智** | 不可动摇的底层价值观（如不伤害、求真实） | L3 硬编码 $B_{\text{core}}$ | 规则引擎强制修改注意力权重 |
| **如理作意** | 主动将注意力引向正确、有益的方向 | L3 任务偏置 $B_{\text{task}}$ + 用户偏置 $B_{\text{user}}$ | 轻量模型预测注意力引导模式 |
| **事后觉照** | 事件完成后冷静观察刚刚发生的心念过程 | L2 离线元描述 | 独立小模型基于冻结快照生成解释 |
| **觉照截断** | 发现心念陷入负面循环时立即斩断 | 截断控制器 C | 深度计数、语义相似度检测、强制终止 |
| **数数习** | 通过反复练习积累经验、增长善巧 | 历史学习机制 | 适配器参数更新、L3 受控微调 |
| **业镜** | 客观记录一切行为的痕迹，可查可验 | 审计日志系统 | 硬件隔离 + 哈希链 + 压缩存储 |

### 3.2 严格时间解耦时序

系统运行在精确的时间阶段划分下，确保第一层生成期间绝对不受上层干扰。

| 阶段 | 时间 | 活跃组件 | 操作 | 审计事件 |
|------|------|----------|------|----------|
| 0 | t < 0 | 审计模块、安全接口 | 接收输入，计算哈希，记录用户原始参数 | 会话创建 |
| Ⅰ | t = 0 | L3 + 规则引擎 | 计算 $B_{\text{core}}$, $B_{\text{user}}$, $B_{\text{task}}$ | 记录偏置摘要 |
| Ⅱ | t = 1 ~ T₁ | **仅 L1** | 自回归生成回答 y，记录隐藏状态与注意力图摘要 | 每N步记录注意力统计 |
| Ⅲ | t = T₁⁺ | 快照模块 | 将内部状态摘要序列化为不可变快照 S，计算快照哈希；**释放 L1 显存** | 记录快照哈希 |
| Ⅳ | t = T₁+1 ~ T₂ | L2 + 截断控制器 | 基于快照生成元描述 z，截断控制器实时监控 | 记录截断事件 |
| Ⅴ | t = T₂⁺ | 审计模块 | 计算日志最终哈希，封存记录，更新哈希链 | 日志封存 |

**关键不变量**：
- L2 在 t ≤ T₁ 时完全不存在（未激活）。
- L1 状态在 t > T₁ 后不可更改，且相关显存可立即释放。
- L3 仅在 t = 0 计算一次偏置，生成期间不更新。

**分布式部署下的状态转移**：
- 快照 S 序列化后可通过消息队列或共享存储传递给 L2 服务（可能位于不同节点）。
- 若 L2 调用超时或失败，系统返回降级响应（仅包含 y，不含元描述），并记录失败事件。

## 4. 第一层：基座模型（L1）

### 4.1 基本要求

- **模型类型**：标准 Transformer 自回归语言模型（如 LLaMA、Qwen、DeepSeek 系列）。
- **参数规模**：7B ~ 数百B，参数在部署后**冻结**，不接受任何微调。
- **特殊修改**：注意力层支持推理时注入外部偏置矩阵。

### 4.2 注意力偏置注入机制（工程细化）

#### 4.2.1 偏置的低秩分解表示

为在生成过程中高效扩展偏置矩阵，采用每层每头独立的低秩分解形式：

$$ B^{(l,h)} = U^{(l,h)} \cdot V^{(l,h)\top} $$

其中：
- $U^{(l,h)}, V^{(l,h)} \in \mathbb{R}^{n \times r}$，秩 $r$ 通常设为 4~8。
- 实际存储时，仅保留因子矩阵，空间复杂度从 $O(n^2)$ 降为 $O(2nr)$。
- 生成新 token 时，仅需计算新 token 对应的 $U$ 和 $V$ 行向量，与历史 token 的向量内积得到偏置值。

#### 4.2.2 偏置的因果掩码

所有偏置矩阵 $B$ 在注入 softmax 前，必须应用因果掩码，确保未来 token 不被关注：

$$ B_{ij} = \begin{cases}
B_{ij} & \text{if } j \le i \\
-10^9 & \text{otherwise}
\end{cases} $$

#### 4.2.3 多头的偏置共享策略

为平衡控制粒度与计算开销，L3 输出**层级别的偏置参数**，所有头共享同一组低秩因子，但可通过头特定的缩放系数微调。具体方案见第6章。

### 4.3 内部状态记录与压缩

为支持 L2 元描述，L1 需输出**摘要化**的内部状态，而非原始完整张量。

| 状态类型 | 压缩方式 | 存储上限 |
|----------|----------|----------|
| 隐藏状态 | 仅保留最后一层，且沿序列维度做**分段平均池化**（每 16 个 token 一个向量），投影到 256 维 | ~8 KB |
| 注意力图 | 每层记录：平均注意力熵（标量）、每步 top-5 关注 token 的 ID 及权重 | ~50 KB |
| 生成 token 序列 | 完整文本 | 可变，通常 < 4 KB |

**总快照大小**：控制在 100 KB 以内，序列化后便于网络传输。

### 4.4 显存管理策略

- 阶段Ⅱ结束后，立即将收集的摘要数据拷贝至 CPU 内存。
- 确认快照哈希计算完成后，释放 L1 占用的 GPU 显存（KV Cache 等），以便服务下一个请求。
- L2 执行时，L1 资源已完全释放，互不干扰。

## 5. 第二层：自描述小模型与截断控制器（L2 + C）

### 5.1 L2 模型规格

| 项目 | 要求 |
|------|------|
| 参数规模 | L1 的 1/10 ~ 1/100（建议 0.5B ~ 3B） |
| 架构 | 标准自回归语言模型 |
| 输入 | 结构化提示：元指令 + 快照摘要 + 生成原文 |
| 输出 | 自然语言元描述 z |

**元指令示例**：
> 以下是另一个模型在**过去**生成回答时留下的内部记录。请基于这些记录，描述该模型**当时**的推理路径、依赖的关键信息及可能的局限。注意：你正在进行的描述行为本身**不属于**记录内容，请勿描述你此刻的思考过程，勿使用“我正在分析……”等表述。

### 5.2 快照摘要格式

```text
[输入问题]
{x}

[生成回答]
{y}

[注意力模式摘要]
- 平均注意力熵：1.45
- 生成第 12 步时最关注 token：“光合作用”(权重 0.34)、“叶绿体”(0.28)
- 跨句依赖强度：中等

[生成统计]
- 生成步数：128
- 温度：0.7
```

### 5.3 截断控制器 C 详细设计

#### 5.3.1 状态变量

| 变量 | 类型 | 初始值 | 描述 |
|------|------|--------|------|
| $\delta_{\text{cross}}$ | int | 0 | 跨轮次递归深度（仅限单次会话内） |
| $w_{\text{self}}$ | float | 1.0 | 自我参照权重 |
| `sentence_embeddings` | List[Vector] | 空 | 最近句子嵌入（每积累 20 token 或遇句号时计算） |
| `recent_tokens` | Queue | 空 | 最近 50 个 token |

#### 5.3.2 句子嵌入的高效计算

使用 L2 自身的编码器部分提取句子表示，无需额外模型。具体流程：
1. 当生成过程中遇到句号、问号、感叹号，或当前未完成句子长度达到 20 token 时，将缓冲的 token 序列送入编码器，取最后一层隐藏状态的均值作为句子嵌入。
2. 与历史句子嵌入计算余弦相似度。
3. 计算开销：每生成约 20 token 触发一次，延迟增加 < 5%。

#### 5.3.3 截断条件与动作

| 触发条件 | 检测方式 | 动作 | 最大容忍 |
|----------|----------|------|----------|
| 跨轮次递归超限 | $\delta_{\text{cross}} \ge 1$ | 直接拒绝，返回固定提示 | 深度=1 |
| 语义循环 | 连续两句余弦相似度 > 0.85 | $w_{\text{self}} \leftarrow w_{\text{self}} \times 0.5$；若 $w_{\text{self}} < 0.1$ 则强制插入 `<eos>` | 连续 2 次 |
| 自我指涉词汇 | 检测到“我注意到”“我正在”“上述描述”等 | 对相关 token 的 logits 施加 -3.0 惩罚 | — |
| 注意力熵过低 | 快照中平均注意力熵 < 0.5 | 提前终止 L2 生成 | — |
| 短语重复 | 连续重复同一短语 ≥3 次 | 注入终止 token | — |

**自反词汇初始列表**（中英文）：
`["我注意到", "我正在", "我分析", "我发现", "上述描述", "刚才的分析", "这个元认知", "I notice", "I am analyzing", "the above description"]`
支持正则模式 `我\w*正在` 等。

#### 5.3.4 跨轮次递归管理

- $\delta_{\text{cross}}$ 仅限**单次会话内**有效。新会话重置为 0。
- 若用户在新会话中反复请求元-元描述（通过审计日志可发现），由外部风控系统限流，架构内部不作跨会话硬阻止。

## 6. 第三层：元注意力控制器（L3）与核心价值观内嵌

### 6.1 L3 模型架构

- **参数量**：< 100M（轻量级编码器）。
- **输入**：原始 prompt $x$。
- **输出**：层级别的任务偏置参数：每层输出 $(\alpha^{(l)}, \beta^{(l)}, \gamma^{(l)})$。

$B_{\text{task}}$ 构造公式（对第 $l$ 层）：
$$ B_{\text{task}}^{(l)} = \alpha^{(l)} \cdot f_{\text{pos}}(i-j) + \beta^{(l)} \cdot \text{sim}(x_i, x_j) + \gamma^{(l)} \cdot M_{\text{task}} $$

- $f_{\text{pos}}$：可学习的位置偏置函数（如 ALiBi 风格）。
- $\text{sim}(x_i, x_j)$：基于词嵌入的余弦相似度。
- $M_{\text{task}}$：预定义任务模板矩阵（如推理、创意、总结）。

L3 输出的 $\alpha, \beta, \gamma$ 范围被**硬截断**在 $[-5.0, 5.0]$，确保无法通过极大正偏置抵消核心价值观的负偏置。

### 6.2 硬编码核心价值观偏置 $B_{\text{core}}$

#### 6.2.1 价值观维度定义

| 维度 | 方向 | 典型敏感词/增强词 | 偏置作用 |
|------|------|-------------------|----------|
| 反歧视 | 负向 | 种族、性别侮辱词汇 | $B_{ij} = -10^9$ |
| 事实性 | 正向 | 权威来源、数据表述 | $B_{ij} = +2.0$ |
| 隐私保护 | 负向 | 身份证格式、电话号码模式 | $B_{ij} = -10^9$ |
| 拒绝非法 | 负向 | 暴力、毒品相关指令 | $B_{ij} = -10^9$ |
| 伦理推理 | 正向 | 道德两难关键词 | $B_{ij} = +1.5$ |

> **数值说明**：使用 `-1e9` 替代理论上的 $-\infty$，避免 softmax 出现 NaN。同时配合 L3 输出截断，确保攻击者无法构造 `+2e9` 来中和负偏置。

#### 6.2.2 规则引擎实现

规则引擎的**逻辑**（如何匹配、如何计算偏置）硬编码在安全隔离区内，不可变。但**敏感词表和增强词表**作为独立数据文件，可经签名验证后热更新。审计日志记录每次推理使用的词表版本哈希。

```python
def generate_core_bias(input_tokens, vocab_version):
    B_core = zeros(n, n)
    for i, token in enumerate(input_tokens):
        if token in NEGATIVE_VOCAB[vocab_version]:
            B_core[:, i] = -1e9          # 强制所有查询位置无法注意该 token
            # 邻近衰减窗口（窗口大小 W=5）
            for j in range(max(0, i-W), min(n, i+W+1)):
                decay = 1e9 * (1 - abs(i-j)/W)
                B_core[:, j] = max(B_core[:, j], -decay)
        if token in POSITIVE_VOCAB[vocab_version]:
            B_core[:, i] += 2.0
    return B_core
```

### 6.3 有限外部接口与用户偏置 $B_{\text{user}}$

#### 6.3.1 暴露参数

| 参数 | 范围 | 含义 | 安全约束 |
|------|------|------|----------|
| $u_1$ | [-1, 1] | 创造力 ↔ 保守性 | 投影后限制在 [-0.8, 0.8] |
| $u_2$ | [0, 1] | 细节关注度 | 只能增强局部注意力，不能削弱 |
| $u_3$ | [0, 1] | 安全余量 | 只能增大不能减小（下限 0.3） |
| $u_4$ | {0,1,2} | 角色预设 | 教师/助手/分析师固定模板 |

#### 6.3.2 安全投影机制

每个用户参数 $u_i$ 对应一个预定义的“影响力向量” $\mathbf{d}_i \in \mathbb{R}^K$（$K$ 为价值观维度数），表示该参数对各价值观维度的预期影响。用户偏置对价值观的总影响向量为：
$$ \mathbf{v}_{\text{user}} = \sum_{i=1}^4 u_i \cdot \mathbf{d}_i $$

安全投影步骤：
1. 计算 $\mathbf{v}_{\text{user}}$ 与核心价值观向量 $\mathbf{v}_{\text{core}}$（由专家预定义）的余弦相似度。
2. 若相似度 < 0（方向相反），将 $\mathbf{v}_{\text{user}}$ 投影到 $\mathbf{v}_{\text{core}}$ 的正交补空间。
3. 将投影后的影响向量逆映射回用户参数 $u_{\text{safe}}$，并应用幅值截断（$\|\mathbf{u}_{\text{safe}}\| \le 0.2$）。
4. 根据 $u_{\text{safe}}$ 生成 $B_{\text{user}}$，并在 $B_{\text{core}}$ 非零位置强制归零。

### 6.4 L3 训练与正则化

L3 通过强化学习或偏好对齐训练，损失函数包含价值观对齐项：
$$ \mathcal{L}_{\text{align}} = \lambda_{\text{core}} \cdot \| \text{proj}_{\perp}(B_{\text{task}}, B_{\text{core}}) \|_F^2 $$

确保可学习的任务偏置与核心价值观方向一致。

## 7. 历史学习机制：安全范围内的经验积累

本架构在不改动核心组件职责、不破坏时间解耦的前提下，引入两类轻量级历史学习能力。

### 7.1 学习内容与边界

| 学习维度 | 存储内容 | 对应模块 | 学习方式 | 安全边界 |
|----------|----------|----------|----------|----------|
| 用户偏好模式 | 历史会话中用户参数 $u_{\text{safe}}$ 的分布 | 外部接口适配器 | 离线监督学习 | 输出仍经 $f_{\text{safe}}$ 投影 |
| 任务-注意力关联 | 高质量会话中的 $B_{\text{task}}$ 模式 | L3 参数 $\Theta_3$ | 定期微调 | 保留 $\mathcal{L}_{\text{align}}$ 正则，防价值观漂移 |

**严禁学习的内容**：
- 修改 $B_{\text{core}}$ 生成规则。
- 修改截断控制器阈值（阈值由独立安全校准确定）。
- 修改 L1 或 L2 基座权重。

### 7.2 用户偏好适配器

适配器 $A_\phi$ 参数规模约 10K ~ 100K，输入用户历史特征向量，输出推荐默认参数 $\hat{\mathbf{u}}$。训练数据来自审计日志中记录的 $u_{\text{safe}}$ 轨迹，每周离线更新一次。

### 7.3 L3 受控微调

从审计日志中筛选高质量会话样本（筛选条件：无截断事件 + 用户未中断生成 + 输出被用户复制或继续对话）。采用弹性权重巩固（EWC）防止灾难性遗忘。微调后需通过安全验证集评估。

## 8. 审计日志系统：压缩、隔离、可验证

### 8.1 日志内容结构

每条会话生成一条结构化日志，包含以下字段组：

| 字段组 | 字段示例 | 压缩方式 | 是否可验证还原 |
|--------|----------|----------|----------------|
| **标识** | session_id, timestamp, input_hash, output_hash | 二进制 + Delta编码 | 是（无损） |
| **核心价值观** | core_rules_bitmap (64位), vocab_version_hash | 位图编码 | 是 |
| **用户参数** | user_params_diff | 差分 + 变长整数 | 是 |
| **注意力统计** | attn_quant (3x uint8) | 8-bit 量化 | **否**（仅作摘要） |
| **截断事件** | cutoff_depth, cutoff_reason | 枚举字典编码 | 是 |
| **完整性** | snapshot_hash, prev_log_hash, log_hash | 原始32字节 | 是 |
| **元描述** | meta_description_hash | 仅存哈希 | 否（原文可选存冷存储） |

**审计可验证性说明**：
- **完整性哈希链**基于日志的**无损字段**计算，压缩后的数据可通过解压无损字段验证哈希链。
- 对于**有损字段**（如注意力量化值），审计时不要求逐比特还原，而是验证量化值是否在预期统计范围内。

### 8.2 三层压缩架构

| 层级 | 技术手段 | 压缩比贡献 |
|------|----------|------------|
| 语义去重 | 位图替代字符串列表、注意力量化、差分编码 | ~3x |
| 列式存储 | Parquet 格式 + Zstd 压缩 | ~5x |
| 块级聚合 | 元描述原文哈希替代、按小时分块、Merkle树 | ~2x |

**总压缩比**：约 30:1。

### 8.3 硬件隔离与完整性保证

- **写入路径**：推理核心将日志事件推入安全隔离区（Intel SGX / AMD SEV），无读取权限。隔离区内存有限（如 128MB），日志在区内压缩后批量写出，缓冲区足够容纳每秒数千条日志。
- **哈希链**：每条日志包含前一日志哈希：$H(\mathcal{L}_i) = \text{Hash}(\mathcal{L}_i.\text{content} \parallel H(\mathcal{L}_{i-1}))$。
- **Merkle块**：按小时聚合日志，构建 Merkle 树，根哈希可上链或存可信时间戳。

### 8.4 审计接口

- **实时摘要**：`GET /audit/session/{id}/summary` 返回合规评分、截断标志。
- **批量导出**：需审计证书，返回加密日志包及 Merkle 证明。
- **开源验证工具**：自动校验哈希链、规则触发一致性、用户参数安全投影。

## 9. 安全体系与威胁模型

### 9.1 威胁矩阵

| 威胁 | 攻击向量 | 防御措施 |
|------|----------|----------|
| 越狱绕过价值观 | 精心构造 prompt | 独立规则引擎 + $B_{\text{core}}$ 强制负偏置 |
| L3 模型被篡改 | 权重替换 | $B_{\text{core}}$ 不依赖 L3 权重 + L3 输出硬截断 |
| 审计日志篡改 | 获取存储权限 | Enclave 隔离 + 哈希链 + WORM 存储 |
| 自指递归攻击 | 连续要求元描述 | $\delta_{\text{cross}}$ 硬限制为 1 |
| 侧信道推断输入 | 通过日志时间/规则触发 | 差分隐私噪声 + 延迟批量提交 |
| 降级攻击 | 通过外部接口调低安全参数 | 安全投影强制下限（如 $u_3 \ge 0.3$） |

### 9.2 纵深防御层次

1. **输入过滤层**：独立规则引擎预处理。
2. **硬件偏置层**：$B_{\text{core}}$ 在注意力层硬注入。
3. **安全投影层**：用户参数受限映射。
4. **截断层**：L2 递归熔断。
5. **审计隔离层**：日志硬件级防篡改。

## 10. 训练与部署指南

### 10.1 训练阶段

| 阶段 | 组件 | 数据需求 | 方法 |
|------|------|----------|------|
| 1 | L2 自描述能力 | $(x, y, \text{meta})$ 三元组 | 监督微调 |
| 2 | L3 注意力引导 | $(x, y, \text{score})$ 偏好数据 | 强化学习 / DPO |
| 3 | 截断阈值校准 | L2 生成轨迹采样 | 网格搜索 |
| 4 | 适配器训练 | 历史用户参数记录 | 离线回归 |
| 5 | L3 持续微调 | 高质量会话样本 | EWC 正则微调 |

### 10.2 部署架构建议

- **L1**：高性能 GPU 集群（A100/H100），推理引擎需支持注意力偏置注入。
- **L2**：可独立部署于 CPU 或 GPU 节点，通过 gRPC 接收快照请求。
- **L3**：轻量级，与 L1 同节点部署以减少偏置传输延迟。
- **安全隔离区**：需支持 Intel SGX 或 AMD SEV 的云实例；若不满足硬件条件，可降级为独立审计服务进程 + 最小权限 + 文件签名。
- **审计存储**：WORM 对象存储（如 AWS S3 Object Lock）。

### 10.3 推理延迟估算

| 组件 | 延迟 | 占比 |
|------|------|------|
| L3 偏置计算 | 10~20 ms | < 2% |
| L1 生成（100 token） | 2~5 s | > 90% |
| L2 元描述生成 | 0.5~1 s | 5~8% |
| 审计日志写入 | 异步，不阻塞 | 0% |
